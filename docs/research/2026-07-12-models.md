## Decision summary

Use `google/gemma-4-E2B-it` as Neko's default offline dialogue/reasoning model,
served from the official LiteRT-LM mobile artifact. Keep
`nvidia/Nemotron-Labs-Audex-2B` installed on disk for evaluation, but do not make
its full audio path boot-resident on this 8 GB Orin Nano.

This is not merely a preference:

- Gemma 4 E2B has Apache-2.0 terms, first-party quantized artifacts, tool-use
  support, and a published LiteRT-LM measurement on Jetson Orin Nano.
- Audex is governed by NVIDIA's OneWay Noncommercial License, is only days/weeks
  into public availability, has no official quantized checkpoint, and its base
  BF16 checkpoint plus released FP32 streaming speech decoder contain about
  8.26 GB of weights before KV cache, activations, CUDA graphs, or runtime state.
  It cannot fit as a complete unquantized speech service in 7.3 GiB usable unified
  memory.
- Audex's released TTS is fixed-voice and its speech-to-speech demo is cascaded.
  It does not remove the need for a character-specific TTS/sound design.

Recommended boot profiles:

| Profile | Boot-resident model | Other workers | Purpose |
| --- | --- | --- | --- |
| `neko-local` | Gemma 4 E2B LiteRT-LM | wake/VAD/ASR, TTS, audio scheduler, metric proximity | Default offline/online assistant |
| `neko-depth-lab` | Gemma 4 E2B or no LLM | ZipDepth plus detector/tracker | Perception profiling |
| `neko-audex-lab` | Audex only, shortest practical context | Minimal audio I/O; stop competing GPU models | Explicit research/A-B tests |
| `neko-degraded` | No generative LLM required | wake word, deterministic commands, prerecorded sounds/stories | Low-memory/failure mode |

"Installed at boot" should mean the supervisor and selectable profile are
available immediately. It should not mean both large models are simultaneously
resident. The owner has confirmed this interpretation; Gemma is normal and Audex
is stopped/selectable.

  `9dbdf8a839e4e9e0eb56ed80cc8886661d3817cf`

- Architecture: about 2.3B effective parameters and about 5.1B total parameters

### Local Jetson result on 2026-07-13

The pinned artifact was downloaded, checksummed, imported, and tested with
LiteRT-LM 0.14.0 on this R39.2.0 Orin Nano. CPU text generation and CPU audio
input both work. A short 256-token-prefill/64-token-decode CPU run with six
threads and a 2,048-token cache measured 37.30 prefill tokens/s, 14.51 decode
tokens/s, 1.1278 s initialization, and 6.9330 s TTFT. After the owner's desktop
logout and all download work ended, a vendor-shaped 1,024-prefill/256-decode run
with the GDM greeter still resident measured
142.28 prefill tokens/s, 14.73 decode tokens/s, 1.8600 s initialization, and
7.2651 s TTFT. Its observed peak was 2,249/7,485 MB RAM, about 50.2 C and 8.51 W,
with 0% GPU use. The different prefill rates reflect optimized batch/signature
shape, not a contradiction or a short-prompt guarantee. The model also correctly
transcribed the ALSA `Front_Center.wav` sample as `front center` and exited
cleanly in about 3.2 seconds. These are one-run smoke/baseline results, not
production latency or quality claims.

The prebuilt GPU backend generated the requested `meow`, then failed in
NVVM/Dawn/Vulkan pipeline creation and exited 139. The observed failure was not
an OOM or thermal event. It is consistent with upstream Jetson reports
<https://github.com/google-ai-edge/LiteRT-LM/issues/2570> and
<https://github.com/google-ai-edge/LiteRT-LM/issues/2338>. Accordingly, the
published Orin GPU numbers above are not reproducible on the current software
stack, and the stock GPU backend must not be used for the boot service. CPU is
the measured safe baseline while a fixed upstream build, source build, or the
official GGUF/llama.cpp fallback is evaluated.

LiteRT-LM 0.14.0's stock OpenAI-compatible server does not expose backend,
thread-count, or cache-size controls. Its implementation lazily loads the
artifact defaults with an unspecified maximum token count. A bounded local
server wrapper is required before autostart so the 8 GB board cannot silently
reserve an unsuitable context.

### Alternative Gemma artifacts to retain as fallbacks

- Official QAT Q4_0 GGUF:
  <https://huggingface.co/google/gemma-4-E2B-it-qat-q4_0-gguf>. The language
  model file is 3,349,514,112 bytes; the multimodal projector is an additional
  986,833,312 bytes. This is the portable llama.cpp/CPU-offload fallback.
- Official W4A16 compressed tensors:
  <https://huggingface.co/google/gemma-4-E2B-it-qat-w4a16-ct>. The inspected
  repository is about 8.35 GB, so "W4A16" does not imply a sub-4-GB file here.
- Official mobile compressed tensors:
  <https://huggingface.co/google/gemma-4-E2B-it-qat-mobile-ct>.
- MTP assistant/draft model:
  <https://huggingface.co/google/gemma-4-E2B-it-assistant>. Do not enable
  speculative decoding until baseline quality, memory, and latency are recorded.
- vLLM recipe for larger NVIDIA deployments:
  <https://docs.vllm.ai/projects/recipes/en/stable/Google/Gemma4.html>.

LiteRT-LM remains the first CPU/multimodal baseline because it works locally and
has the most relevant first-party edge artifact. Its version 0.14 native server
needs a bounded wrapper, and its prebuilt GPU path is currently unusable here.
The NVIDIA memory skill's generic Orin recommendation is llama.cpp/Q4_K_M; the
official Gemma Q4_0 GGUF is therefore the next GPU/offload fallback to benchmark,
not a claim that its quality or memory use will be better. Retain LiteRT CPU for
audio input while comparing the fixed/source LiteRT path and GGUF path.

Audex accepts audio and text and can perform audio understanding, ASR,
translation, text reasoning, TTS, general audio generation, and cascaded
speech-to-speech. It supports thinking and instruct modes. Its released TTS has
a fixed voice; the paper indicates speaker prompting was erased during later
training. Cat persona therefore still needs separate TTS/prosody and curated
meow/purr assets.

### Released storage and memory facts

The inspected Hugging Face repository totals 12,281,588,752 bytes. Important
components include:

| Component | File bytes | Role |
| --- | ---: | --- |
| Full checkpoint shard 1 | 4,500,748,032 | BF16 multimodal model |
| Full checkpoint shard 2 | 1,301,260,840 | BF16 multimodal model |
| Causal speech decoder | 2,455,045,520 | FP32 streaming speech decoder |
| Enhancement VAE | 2,622,247,900 | Optional 48 kHz general-audio enhancement |
| Bundled NV-Whisper weights | 1,273,988,176 | Audio encoder/preprocessor component |

The full checkpoint plus causal speech decoder alone are about 8.26 GB of
weight files. The process additionally needs tokenizer/config state, KV cache,
activations, CUDA context, graphs/workspaces, audio buffers, and application
memory. The board exposes only 7.3 GiB usable RAM. No swap was configured at
inspection, and swap would not make real-time GPU residency viable anyway.

### Official runtime and missing optimization evidence

The upstream README specifies:

- `vllm/vllm-openai:v0.20.0-cu129` (vLLM 0.20.0/CUDA 12.9).
- Custom Nemotron dense and audio-QA plugins installed from the model repository.
- `vllm[audio]` extras for codecs.
- Transformers 4.53.3 as a tested Hugging Face path, with newer 5.x claimed to
  work.
- Remote/custom model code. Any deployment must pin and review that code.

No official AWQ, GPTQ, NVFP4, GGUF, LiteRT, ONNX, TensorRT, or MLX Audex
checkpoint was found. NVIDIA provides no published Audex TTFT, real-time factor,
tokens/s, power, minimum VRAM, concurrency, or Jetson benchmark. Non-NVIDIA
operation is not validated. Speech requires roughly 50 autoregressive model
steps per second of output audio; general audio requires roughly 200, so model
size alone does not establish real-time behavior.

### Audex deployment decision

1. **Completed:** the exact repository was downloaded and checksummed after the
   owner confirmed personal/noncommercial eligibility.
2. Keep it out of the normal boot-resident set on this 8 GB host.
3. Evaluate the ASR-only/full-input path and TTS path separately with a very
   short context with the host headless and competing workers stopped.
4. No remote NVIDIA machine is available in this revision. If that changes in a
   future revision, 24 GB is a sensible experimental floor and 32 GB gives safer
   coexistence; these are engineering recommendations, not NVIDIA minimums.
5. Treat any third-party quantization as an experiment requiring audio/text
   quality regression tests and a new artifact/provenance record.

## Required local benchmark matrix

Every run must record model revision, artifact checksum, runtime commit/version,
backend, quantization, prompt/audio lengths, context limit, power mode, clocks,
temperature, other resident workers, and cold/warm state.

| Test | Required outputs |
| --- | --- |
| Gemma text, cold | load-to-ready, first-token latency, peak RAM, failure/retry behavior |
| Gemma text, warm | prefill tok/s, decode tok/s, P50/P95 TTFT, 8K conversation memory |
| Gemma audio input | 1/5/15/30 s clips, transcription/intent quality, encoder latency, peak RAM |
| Gemma tool calls | valid-schema rate, false tool calls, constrained local-command suite |
| Gemma + perception | latency and memory with detector/tracker and ZipDepth/metric sensor workers |
| Gemma + full audio | end-of-speech to first audible response, barge-in, AEC leakage, peak temperature/power |
| Audex ASR/input only | startup fit, WER/intent accuracy, TTFT, peak RAM, no decoder loaded if possible |
| Audex TTS | real-time factor, first-audio latency, decoder memory, intelligibility, fixed-voice suitability |
| 30-minute soak | P95/P99 latency, throttling, OOM/restarts, memory leaks, audio underruns |

Enabling a bounded experimental unit is not the same as accepting a production
boot profile. Do not call a profile boot-validated until it completes cold-boot,
health-check, warm-up, restart, network-loss, and combined soak tests.
