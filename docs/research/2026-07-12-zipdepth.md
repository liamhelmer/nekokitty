## Bottom line

ZipDepth is a very small, fast, current monocular relative-depth model. It is
not a metric range sensor and must not own greeting distance thresholds,
approach speed, navigation, braking, or collision safety.

It predicts affine-invariant inverse depth from one RGB frame. The published
AbsRel evaluation aligns every prediction to ground-truth scale and shift before
scoring. The paper explicitly lists temporal flicker and metric-depth support as
future work. A person can look nearer than another person within a frame, but a
raw ZipDepth value does not mean “1.8 metres.”

Keep ZipDepth because the owner explicitly requested it and because it is useful
for visual effects, relative scene structure, foreground ordering, and an
experimental secondary cue. Pair it with physical metric sensing for actual
proximity behavior.

## Model behavior

- Input: one RGB image. No camera intrinsics, depth sensor, IMU, or prior frames.
- Output: a positive inverse-depth image at model-input resolution.
- Default preprocessing: BGR to RGB, normalize to `[0,1]`, resize the shortest
  side to 384 while retaining aspect ratio, and round dimensions to multiples of
  32. A 16:9 frame is therefore roughly 672x384, not 384x384.
- Fused inference model: about 6.14M parameters and 3.0 GMAC at 384x384.
- Training: scale-and-shift-invariant distillation from Depth Anything V2-Large
  using roughly 14.1M images from 17 domains.
- The repository has no person detector, segmenter, confidence head, tracker,
  temporal filter, camera daemon, ROS node, or live deployment service.
- Repository “video” processing applies the image model independently per frame.

Known/acknowledged weaknesses include flicker, thin structures, fine boundaries,
repetitive or textureless regions, strong reflections, over-smoothing, and
incorrect ordering. In a moving cart, pitch/ego-motion and changing scale/shift
make metre thresholds even less defensible.

## License/provenance warning

The repository says MIT and includes both weights. It does not provide a separate
checkpoint license or model card. The paper says the student was distilled from
Depth Anything V2-Large, whose official project licenses the Large weights under
CC-BY-NC-4.0 while licensing only its Small model under Apache-2.0:
<https://github.com/DepthAnything/Depth-Anything-V2#license>.

That does not by itself prove the student inherits the teacher's terms, but the
ZipDepth repository does not resolve the discrepancy. Before public/commercial
deployment, obtain written clarification from the authors or legal advice. The
research copy may be evaluated separately from a shipping decision.

## Performance evidence and caveats

These are author-reported median model-forward measurements after warm-up, batch
1. They exclude the rest of a camera/detector/tracker/decision pipeline.

| Platform | Input/path | Reported model FPS |
| --- | --- | ---: |
| Orin NX 16GB, 15W | 384x384 fused FP32 | 34 |
| Orin NX 16GB, 15W | 384x384 fused FP16 | 42 |
| Orin NX 16GB, 15W | 384x384 TensorRT FP16 | 77 |
| Orin NX 16GB, 15W | 384x672 fused FP32 | 24 |
| Orin NX 16GB, 50W | 384x384 TensorRT FP16 | 196 |
| Intel i7-11800H | TorchScript frozen | 48.9 |
| iPhone 12 ANE | CoreML FP16 NPU path | 375 |
| Poco X3 | TFLite GPU FP16 | 16 |

At 384x384 the reported RTX peak model allocation is roughly 104 MB for fused
FP32 or 77.5 MB for fused FP16. A complete PyTorch/TensorRT process uses more.
There is no Orin Nano result and no result under simultaneous LLM/ASR/TTS load.
The model should fit easily; shared memory bandwidth and GPU scheduling are the
real questions.

## Jetson deployment path

1. Vendor the exact source commit and `zipdepth_base.pth`; verify its checksum.
2. Use the host's JetPack-matched PyTorch for export only. Do not install the
   repository's whole training requirements file on the target.
3. Call `fuse_for_inference()` and perform remaining Conv-BN fusion.
4. Use the official static-shape ONNX opset-17 export script:
   <https://github.com/fabiotosi92/ZipDepth/blob/main/scripts/export.py>.
5. Deliberately choose the deployed camera geometry. For 16:9, build a
   384x672 engine or document crop/letterbox and coordinate transforms.
6. Build a fixed-shape FP16 TensorRT engine on this Jetson and its exact
   TensorRT version. No engine is portable across arbitrary TensorRT/GPU builds.
7. Name/cache the engine by source commit, checkpoint checksum, TensorRT version,
   GPU, precision, and input dimensions. Build during provisioning, never boot.
8. Preload/warm the engine, then publish readiness.
9. Run person detection/tracking separately and associate relative depth only
   with stable person ROIs/tracks.
10. Benchmark the entire perception graph with the audio/LLM profile active.

Do not use `torch.compile` on the immediate-boot path because first compilation
can be long. No INT8 accuracy/performance or calibration recipe is published;
FP16 is the defensible first TensorRT precision.

The upstream README and requirements currently disagree about the minimum
PyTorch version. The source requirements also include training dependencies.
Production should use a minimal TensorRT runtime, not a mutable Python training
environment.

## Recommended physical proximity system

Best off-the-shelf fit: a Luxonis OAK stereo/ToF spatial camera that performs
person detection, metric depth, and persistent tracking on-device, sending only
anonymous track metadata and XYZ millimetres to the Jetson.

- Spatial perception:
  <https://docs.luxonis.com/overview/toplevel-features/spatial_perception>
- Object tracker with ID and XYZ/velocity:
  <https://docs.luxonis.com/software-v3/depthai/examples/object_tracker/object_tracker>
- Depth accuracy guidance:
  <https://docs.luxonis.com/hardware/platform/depth/depth-accuracy/>
- OAK4-D Pro:
  <https://docs.luxonis.com/hardware/products/OAK%204%20D%20Pro>
- Vibration/environmental information:
  <https://docs.luxonis.com/hardware/platform/environmental-specifications/vibration-tolerance>

OAK4-D Pro offers IR assistance and more rugged characteristics but averages
roughly 10–15 W and can peak near 25 W. Older OAK-D/RVC2 devices are typically
closer to 3–5 W. Final choice depends on range, weather, light, vibration, and
power.

Complementary presence/low-light sensing can use TI mmWave:

- People tracking/counting reference design:
  <https://www.ti.com/tool/TIDEP-01000>
- Safer AMR/AGV radar reference design:
  <https://www.ti.com/tool/TIDA-010281>

If perception must stay on Jetson, a JetPack-matched NVIDIA PeopleNet detector
and DeepStream NvDCF tracker can be fused with stereo/ToF/radar. DeepStream 9 is
not the Orin target; select a release explicitly compatible with this L4T R39
host or isolate perception on supported hardware.

## Responsibility split

| Layer | Responsibility |
| --- | --- |
| Social perception | person detection, ephemeral stable track IDs, physically metric XYZ |
| Redundant presence | radar/ToF/ultrasonic for darkness, glare, blind areas |
| Optional ZipDepth | relative scene structure, depth effects, nearest-track ranking only |
| Behavior | hysteresis, dwell, approach trend, cooldown, quiet/stationary constraints |
| Cart safety | independent range chain, bumpers/e-stop/watchdog and qualified controller |

The assistant/LLM may phrase an already-approved greeting. It must not decide
whether a track is safe to approach or whether the cart should brake.

## Perception acceptance tests

- Metric error and dropouts across the requested distance band.
- Adults, children, seated/crouched people, groups, pets, partial occlusion.
- Direct sun, dusk/night, rain/fog if in scope, reflective clothing/glass.
- Cart pitch/yaw/vibration, stationary and at every allowed speed.
- Greeting enter/exit hysteresis, dwell, cooldown, false/repeated greetings.
- End-to-end event latency and stale-track timeout.
- 30-minute combined LLM/audio/perception thermal and memory soak.
- Sensor disconnect, occlusion, dirty lens, restart, and degraded-mode behavior.
